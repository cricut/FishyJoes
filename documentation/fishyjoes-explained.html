Annotation-Driven Exports

FishyJoes uses an elegant trick: export directives are written as HTML comments inside Swift doc comments. This keeps them co-located with the code and doesn't require new language features.

Swift Source

/// Represents a design element. /// /// <!-- FishyJoes.export(DesignElement) --> public struct DesignElement { public let id: UUID public let name: String public var width: Double public var height: Double /// <!-- FishyJoes.export(scale) --> public func scale(by factor: Double) -> DesignElement { ... } }

What FishyJoes Sees

// ExportAnnotation parsed from doc comment: ExportAnnotation( kind: .unmodified, // .unmodified | .asReference | .asMethod name: "DesignElement", cSharpName: "DesignElement", // defaults to `name` isOverride: false, noReturn: false, genericOverrides: [:], // [String: BetterType] omitParameters: [], compatibilityOrder: [] ) // Method also exported (has annotation) — Sourcery sees: SourceryMethod( name: "scale", parameters: [ Parameter(name: "factor", typeName: <SourceryTypeName for "Double">, …) ], returnTypeName: <SourceryTypeName for "DesignElement">, isStatic: false, isThrowing: false, isAsync: false )

Annotation Options

<!-- FishyJoes.export(Name) -->
Export a type with an explicit public name for generated bindings.
<!-- FishyJoes.export(methodName) -->
Export a method or property, giving the desired foreign name. A name is always required.
generic: [T: [String]]
Specify concrete substitutions for generic parameters. The parser accepts tokens plus array and dictionary shorthand.
Doc-comment placement
Annotations live inside /// comments so Sourcery captures them with --parseDocumentation and they stay next to the exported symbol.

The BetterType System

Sourcery gives FishyJoes a TypeName model that mixes name strings with optional tuple, array, dictionary, closure, and generic details. FishyJoes converts these into BetterType — a recursive enum that models type trees consistently. Click nodes to explore.

indirect enum BetterType {
▶ .named(Name) // non-generic named type
A non-generic Swift type, identified by its Name — a struct of module: String?, namespace: [String], and name: String.
Examples: .named(.swift("String")), .named(Name(module: nil, namespace: [], name: "UUID")).
Generic parameters like T, U are also stored as .named with no module, and resolved later via the generics map passed to tryResolve.
▶ .generic(base: Name, args: [BetterType]) // parameterised types
The workhorse for every parameterised type. Optional<T>, Array<T>, Dictionary<K,V>, Set<T>, Result<S,F>, Future<T>, and user-defined generics are all .generic values.
Static helpers (.optional(_:), .array(_:), .dictionary(_:_:), …) are sugar that build .generic instances — they are not separate cases.
▶ .tuple([TupleElement]) // labeled tuples
Each TupleElement is { label: String, type: BetterType }. The label is always a String — unlabelled positions get a numeric stand-in ("0", "1", …), which the rendering code recognises by checking Int(label) == nil. Each element type is itself a BetterType, so nested tuples work too.
▶ .function([BetterType], BetterType, isAsync: Bool, isThrowing: Bool) // closure types
Closure/function types: a list of parameter types → a return type, plus isAsync and isThrowing flags that drive codegen (e.g. suspend in Kotlin, async/Promise in TS, exception bridging across the FFI boundary). Used when Swift APIs accept callbacks that need to be marshalled across the language boundary.
▶ .void // no return value
Maps to the no-value forms used by each target: void for C/JS/Dart-style surfaces, Unit for Kotlin, and the runtime's C# unit type.
▶ .selfType // Self in protocol contexts
Represents any Swift type name whose unwrapped name is Self. BetterType preserves it as .selfType so later analysis can decide whether that API shape is supported.
}

Why not just use Sourcery's TypeName?

❌ Sourcery TypeName (awkward)
name: "Optional<Array<UUID>>" // Just a string — you must // parse it yourself to find // the nested structure
✅ BetterType (recursive)
// Raw form — what the enum actually stores: .generic(base: .swift("Optional"), args: [ .generic(base: .swift("Array"), args: [ .named(Name(name: "UUID", module: nil)) ]) ]) // Same value, written with the static helpers: .optional( .array( .named(Name(name: "UUID", module: nil)) ) ) // .optional(_:) → .generic(base: .swift("Optional"), args: [_]) // .array(_:) → .generic(base: .swift("Array"), args: [_]) // .dictionary, .set, .result, .future are all sugar too. // A real tree — pattern match at any depth: if case .generic(let base, let args) = t, base == .swift("Optional"), let inner = args.first { ... }

Runtime Type Marshalling

Generated bindings don't just call Swift — they convert every value across the language boundary. Each runtime implements a Converter protocol to handle this translation.

// FishyJoesCommonRuntime — the shared contract public protocol Converter { // The Swift type this converter handles (defaults to Self) associatedtype SwiftType = Self } // Primitive types are their own converters: extension Bool: Converter {} extension Double: Converter {} extension String: Converter {} // ... etc. // Generic converters wrap element converters: public enum ArrayConverter<E: Converter>: Converter { public typealias SwiftType = [E.SwiftType] } // Actual to/from methods live on NodeConverter, JavaConverter, etc.

⬡ Node Runtime

  • N-API (native) or WASM
  • napi_value ↔ Swift
  • Event loop bridging
  • Promise / async support
  • Buffer / ArrayBuffer

🎯 Java Runtime

  • JNI (Java Native Interface)
  • jobject ↔ Swift
  • jstring, jarray bridging
  • JVM thread safety
  • Coroutine bridge

♟ Iota Runtime (C#)

  • P/Invoke FFI
  • Pointer marshalling
  • Struct layout mapping
  • Task<T> async bridge
  • GC-safe pinned memory

🎱 Iota Runtime (Dart)

  • dart:ffi
  • Pointer<T> bridging
  • Struct alignment
  • Future<T> async bridge
  • Isolate safety

3. NodeConverter   env.argument(at: 0, converter: Double.self) → Swift Double (via Double.fromNode)
4. Swift   DesignElement.scale(by: 2.0) executes natively
