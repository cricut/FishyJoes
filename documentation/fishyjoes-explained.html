Annotation-Driven Exports

FishyJoes uses an elegant trick: export directives are written as HTML comments inside Swift doc comments. This keeps them co-located with the code and doesn't require new language features.

Swift Source

/// Represents a design element. /// /// <!-- FishyJoes.export(DesignElement) --> public struct DesignElement { public let id: UUID public let name: String public var width: Double public var height: Double /// <!-- FishyJoes.export(scale) --> public func scale(by factor: Double) -> DesignElement { ... } }

What FishyJoes Sees

// ExportAnnotation parsed from doc comment: ExportAnnotation( kind: .unmodified, // .unmodified | .asReference | .asMethod name: "DesignElement", cSharpName: "DesignElement", // defaults to `name` isOverride: false, noReturn: false, genericOverrides: [:], // [String: BetterType] omitParameters: [], compatibilityOrder: [] ) // Method also exported (has annotation) — Sourcery sees: SourceryMethod( name: "scale", parameters: [ Parameter(name: "factor", typeName: <SourceryTypeName for "Double">, …) ], returnTypeName: <SourceryTypeName for "DesignElement">, isStatic: false, isThrowing: false, isAsync: false )

Annotation Options

<!-- FishyJoes.export(Name) -->
Export a type with an explicit public name for generated bindings.
<!-- FishyJoes.export(methodName) -->
Export a method or property, giving the desired foreign name. A name is always required.
generic: [T: [String]]
Specify concrete substitutions for generic parameters. The parser accepts tokens plus array and dictionary shorthand.
Doc-comment placement
Annotations live inside /// comments so Sourcery captures them with --parseDocumentation and they stay next to the exported symbol.

The BetterType System

Sourcery gives FishyJoes a TypeName model that mixes name strings with optional tuple, array, dictionary, closure, and generic details. FishyJoes converts these into BetterType — a recursive enum that models type trees consistently. Click nodes to explore.

indirect enum BetterType {
▶ .named(Name) // non-generic named type
A non-generic Swift type, identified by its Name — a struct of module: String?, namespace: [String], and name: String.
Examples: .named(.swift("String")), .named(Name(module: nil, namespace: [], name: "UUID")).
Generic parameters like T, U are also stored as .named with no module, and resolved later via the generics map passed to tryResolve.
▶ .generic(base: Name, args: [BetterType]) // parameterised types
The workhorse for every parameterised type. Optional<T>, Array<T>, Dictionary<K,V>, Set<T>, Result<S,F>, Future<T>, and user-defined generics are all .generic values.
Static helpers (.optional(_:), .array(_:), .dictionary(_:_:), …) are sugar that build .generic instances — they are not separate cases.
▶ .tuple([TupleElement]) // labeled tuples
Each TupleElement is { label: String, type: BetterType }. The label is always a String — unlabelled positions get a numeric stand-in ("0", "1", …), which the rendering code recognises by checking Int(label) == nil. Each element type is itself a BetterType, so nested tuples work too.
▶ .function([BetterType], BetterType, isAsync: Bool, isThrowing: Bool) // closure types
Closure/function types: a list of parameter types → a return type, plus isAsync and isThrowing flags that drive codegen (e.g. suspend in Kotlin, async/Promise in TS, exception bridging across the FFI boundary). Used when Swift APIs accept callbacks that need to be marshalled across the language boundary.
▶ .void // no return value
Maps to the no-value forms used by each target: void for C/JS/Dart-style surfaces, Unit for Kotlin, and the runtime's C# unit type.
▶ .selfType // Self in protocol contexts
Represents any Swift type name whose unwrapped name is Self. BetterType preserves it as .selfType so later analysis can decide whether that API shape is supported.
}

Why not just use Sourcery's TypeName?

❌ Sourcery TypeName (awkward)
unwrappedTypeName: "Array<UUID>" isOptional: true array: Array? // payload if it's an array dictionary: Dictionary? tuple: Tuple? closure: Closure? generic: Generic? // As BetterType.swift puts it: // "Types are trees, not structs full of booleans."
✅ BetterType (recursive)
// Raw form — what the enum actually stores: .generic(base: .swift("Optional"), args: [ .generic(base: .swift("Array"), args: [ .named(Name(name: "UUID", module: nil)) ]) ]) // Same value, written with the static helpers: .optional( .array( .named(Name(name: "UUID", module: nil)) ) ) // .optional(_:) → .generic(base: .swift("Optional"), args: [_]) // .array(_:) → .generic(base: .swift("Array"), args: [_]) // .dictionary, .set, .result, .future are all sugar too. // A real tree — pattern match at any depth: if case .generic(let base, let args) = t, base == .swift("Optional"), let inner = args.first { ... }

Runtime Type Marshalling

Generated bindings don't just call Swift — they convert every value across the language boundary. Runtime-specific converter protocols refine the shared Converter protocol to handle this translation.

// FishyJoesCommonRuntime — the shared contract public protocol Converter { // The Swift type this converter handles (defaults to Self) associatedtype SwiftType = Self } // Primitive types are their own converters: extension Bool: Converter {} extension Double: Converter {} extension String: Converter {} // ... etc. // Generic converters wrap element converters: public enum ArrayConverter<E: Converter>: Converter { public typealias SwiftType = [E.SwiftType] } // Actual to/from methods live on NodeConverter, JavaConverter, IotaConverter, etc.

Object lifecycle — keeping Swift instances alive across the boundary

When a foreign-language caller holds a Swift class instance, the Swift side increments a refcount via env.newRef(). The trick is making sure that refcount is decremented exactly when the foreign wrapper goes away. Each runtime hooks this up differently.

⬡ Node — NAPI reference

  • NodeReference wraps a napi_ref
  • Lifetime tied to the JS object via N-API's reference API
  • Released when the JS wrapper is collected
  • Sources/FishyJoesNodeRuntime/NodeClass.swift

🎯 Kotlin — JNI global ref

  • JavaReference.init → env.globalRef($0)
  • deinit calls DeleteGlobalRef after acquiring the JVM thread
  • Survives across JNI calls (local refs would not)
  • Sources/FishyJoesJavaRuntime/JavaReference.swift

♟ C# — GCHandle + Dispose

  • SwiftReference wraps a GCHandle.Alloc
  • Dispose() → FishyJoesCommonRuntime_AnyBox_releaseRef
  • using var recommended for short scopes
  • c-sharp-runtime/src/SwiftReference.cs

🎱 Dart — Finalizer

  • Finalizer<ffi.Pointer> attached to the Dart wrapper
  • Finalizer callback runs releaseRef when the wrapper is collected
  • No explicit dispose needed
  • dart-runtime/lib/swift_reference.dart

Iota ownership cheat sheet

When values cross the FFI boundary in the Iota runtime (used by C# and Dart), the pointer comes in one of three flavors. Each tells the receiver who is responsible for releasing it.

UnownedRef
Borrowed for the call. The caller still owns it; the callee must not release it. Used for this on instance methods.
CreatedRef
Callee-owned. The caller must consume (= take ownership of) or explicitly release it. Used for return values and out exceptions.
ConsumedRef
Caller-owned at call time, but the callee takes ownership and is responsible for releasing it. Used for arguments-by-value.
Sources: c-sharp-runtime/src/Utilities.cs · Sources/FishyJoesIotaRuntime/Box+Iota.swift

Async bridging — Swift Future<T> across the boundary

A Swift async throws function returns a Future<T> internally. TranslatedFuture picks the right foreign equivalent, and each runtime hops to the foreign event loop before resolving.

Error propagation — Swift throws across the boundary

Swift errors aren't natively representable in any of the target languages, so each runtime turns them into the foreign equivalent. A foreign exception thrown back into Swift is wrapped in IotaException (Iota) or a similar wrapper for the others.

⬡ Node

  1. Swift method throws
  2. rethrowToNode catches it
  3. Calls env.throw(String.toNode(error.description))
  4. Returns nil from the napi_callback
  5. Node sees pending exception → JS reject / throw
Sources/FishyJoesNodeRuntime/Utils.swift

🎯 Kotlin

  1. Swift method throws
  2. JNI exception is set on the env
  3. JVM raises a Java Throwable on return
  4. Inbound Java exceptions are wrapped as JavaThrowableError on the Swift side
Sources/FishyJoesJavaRuntime/JavaConverter+Future.swift

♟ C#

  1. Swift method throws → out CreatedRef exn populated
  2. Generated wrapper calls Check<R>(...)
  3. Check consumes the CreatedRef as a SwiftReference
  4. Throws it as a C# exception
c-sharp-runtime/src/Utilities.cs

🎱 Dart

  1. Swift method throws → OutCreatedRef populated
  2. Generated wrapper calls check((exn) => ...)
  3. check reads the exception ref
  4. Dart throws the unwrapped object
dart-runtime/lib/utilities.dart
Round-trip: a foreign exception caught back in Swift surfaces as an IotaException (Iota) or a per-runtime equivalent, so you can re-raise it on a future throw across the boundary without losing fidelity. See env.newError(Error) in Sources/FishyJoesIotaRuntime/Env.swift.

Call path: TypeScript calls a Swift method

1. JS/TS   const result = element.scale(2.0)
2. N-API callback   Node invokes the generated method callback registered by NodeClass, which enters Swift through FishyJoesNodeRuntime.callbackBody
3. NodeConverter   env.argument(at: 0, converter: Double.self) → Swift Double (via Double.fromNode)
4. Swift   DesignElement.scale(by: 2.0) executes natively
5. NodeConverter   DesignElement.toNode(result, env:) → NAPI.Value (napi_value)
6. JS/TS   The synchronous call returns a fully-typed DesignElement object. Swift async methods use a Promise-returning path.
